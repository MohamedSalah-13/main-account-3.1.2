سأقوم بمراجعة نظام الوردية بالكامل أولاً لفهم البنية الحالية، ثم أقترح إضافات على مراحل.الآن لديّ صورة كاملة عن نظام الورديات الحالي. سأقدم لك مراجعة شاملة مع خطة تطوير متدرجة على مراحل.

## 📋 مراجعة النظام الحالي

### ✅ نقاط القوة
1. **فصل واضح للطبقات**: Controller / Service / DAO / Domain.
2. **التحقق من وجود وردية مفتوحة** قبل الفتح أو الغلق.
3. **تنسيق التاريخ** موحّد عبر `DATE_TIME_FORMATTER`.
4. **تسجيل الأخطاء** عبر `Log4j2`.
5. استخدام **JavaFX Properties** في `UserShift` مما يسهّل الربط مع الواجهة.

## 🚀 خطة التطوير على مراحل

### 🔹 المرحلة 1 — إصلاحات عاجلة (Refactor & Bugfixes)
**الهدف**: تنظيف الكود الحالي بدون تغيير الوظائف.

- [ ] توحيد `isOpen` كـ `BooleanProperty` أو الاحتفاظ به boolean وإزالة التعارض مع `@AllArgsConstructor`.
- [ ] استبدال بناء SQL بـ concatenation بـ **PreparedStatement** في `hasOpenShift`.
- [ ] استخدام `LocalDateTime.now()` من خلال `Clock` قابل للـ inject (لتسهيل الاختبار).
- [ ] استخدام `properties` مباشرة في أعمدة الجدول بدل إنشاء جديدة.
- [ ] تنظيف الاستيرادات المكررة.
- [ ] إضافة **Validation** على الرصيد (>= 0، عدد معقول).
- [ ] تغليف عملية الغلق في `try/finally` وإغلاق الحقول حتى عند فشل الحفظ.

---

### 🔹 المرحلة 2 — ربط الوردية بالحركات المالية
**الهدف**: الوردية يجب أن تعكس الحركة الفعلية، لا أن تكون مجرد طابع زمني.

- [ ] إضافة عمود `shift_id` في جداول:
    - فواتير البيع/الشراء
    - المصروفات
    - حركات الخزينة/العملاء
- [ ] عند فتح الوردية: حفظ الـ `shiftId` في `LogApplication.currentShift`.
- [ ] كل إدخال حركة جديدة يُلصق به `shiftId` تلقائياً.
- [ ] في `closeShift`: حساب:
    - مجموع المبيعات النقدية
    - مجموع المرتجعات
    - مجموع المصروفات
    - **الرصيد المتوقّع** = `openBalance + مبيعات - مرتجعات - مصروفات`
    - **الفرق** = `closeBalance - الرصيد المتوقّع` (عجز/زيادة)
- [ ] حفظ هذه الإجماليات في أعمدة جديدة بالـ `user_shifts`:
```
total_sales, total_returns, total_expenses, expected_balance, difference
```
